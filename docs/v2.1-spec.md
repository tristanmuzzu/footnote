# footnote v2.1 design spec (proposal, not yet built)

Status: **proposal**. v2.0 (ambient spaced repetition) is shipped. This spec defines v2.1.

## One-line goal

Shrink the agent's per-turn footprint to almost nothing and make naming drift
impossible to break the system: the agent only **emits `Learn next:` lines** and
**runs the session-start review**, while deterministic hook code does ALL log
I/O, dedup, and scheduling.

## Explicitly out of scope (by request)

- **No relevance-aware review.** The review surfaces whatever is due, in due order. The agent does not pick terms by today's topic.
- **No knew/fuzzy signal of any kind** — neither user-typed ("I knew it" / "I blanked") nor agent-inferred. Advancement is **purely exposure-based**: a term advances one stage each time it is surfaced for review, and graduates after the last stage. The system trusts the user to recheck a term on their own whenever it resurfaces.

These keep v2.1 lean and keep the original philosophy: footnote points; you look it up yourself, on your own schedule.

## What changes from v2.0

| | v2.0 (now) | v2.1 (this spec) |
| --- | --- | --- |
| Who writes the log | the **agent** (appends new terms, moves graduated ones) | the **hooks only**; the agent never reads or writes the log |
| New-term capture | agent appends to `## Seen once` | a **Stop hook** harvests `Learn next:` lines from the finished reply and writes them |
| Dedup | agent is told to "glance at the log first" | **deterministic normalize + fuzzy-merge** in the hook; the agent can name loosely |
| Agent per-turn job | emit hints + maintain the log | **emit hints + run the review. Nothing else.** |

## Architecture

Two deterministic hooks own everything. The agent touches no footnote file.

### 1. SessionStart hook (scheduler / injector) — extends v2.0
Unchanged selection/advancement logic, plus it becomes a **writer of the markdown log** (for graduations), alongside the harvester:
- Reads the log + `schedule.json`, reconciles, picks <=3 due terms, advances them, graduates ones past the last stage (moves `## Seen once` -> `## Learned`).
- Injects: the terse rules + the <=3 due terms for review. Never the whole log.

### 2. Stop hook (harvester) — NEW
Fires when a reply finishes. Reads the last assistant message from the transcript and:
1. Regex-extracts `Learn next:` entries (the backtick-wrapped `term (tag)` items).
2. For each, computes a **canonical key** and checks it against existing terms.
3. Appends genuinely-new terms to `## Seen once` and schedules them. Skips dupes.
- Atomic writes, rolling backups, append-only enforced in code (see Invariants).

### Agent contract (the whole job)
- Emit `Learn next:` lines per the existing rules (its judgment of what is worth learning).
- Run the session-start review: quiz the injected due terms, no logging, no outcome capture.
- **Never read or write any footnote file.** No "glance to dedup" — the harvester dedups.

Per-turn agent context = the injected rules + <=3 short due-term lines. Flat forever.

## Naming-drift / dedup (the core mechanism)

The agent is bad at exact-string consistency; code is good at it. So matching moves entirely into the hook:
- **Canonical key** = lowercase, strip the `(tag)`, strip punctuation and collapse whitespace. `lockfile (npm)` and `Lock File` both key to `lockfile`.
- **Exact-key match** catches case/tag/spacing variants.
- **Fuzzy match** (normalized token overlap and/or edit distance, conservative threshold) catches near-misses like `transitive deps` vs `transitive dependency`. On a fuzzy hit, treat as the same term (skip the add); record the near-match in a hook log line for transparency, never a silent merge of two genuinely different terms.
- Promotion needs no agent matching at all: the SessionStart hook graduates by canonical key, so loose display naming never blocks promotion.

This is the fix for the exact failure you flagged: "the agent names it slightly differently, so it never moves to Learned."

## Data model + migration (production-grade)

- **Source of truth for which terms exist:** the markdown log (`## Seen once` / `## Learned`), now written ONLY by the two hooks.
- **Scheduling state:** `schedule.json` sidecar (per term: canonical key, stage, timesSeen, lastSeen, nextDue). Rebuildable from the log, so it stays a cache, not precious.
- **Migration v2.0 -> v2.1: none needed.** The markdown format is unchanged (`## Seen once` / `## Learned`). The only change is who writes it (hooks, not the agent) plus the new harvester. Existing logs and `schedule.json` carry over untouched. Zero data risk.

## Invariants (don't regress)

- **Append-only, enforced in code.** A hook write must never reduce the term count; on a would-be decrease it aborts and keeps the backup (fail-safe). This is stronger than v2.0's prose rule, because the writer is now deterministic.
- **Atomic writes** (temp + rename) + **rolling backups** (already in v2.0).
- **The hooks are the only writers.** The agent is read/write-free on footnote files.
- **Bounded injection.** Rules + <=3 due terms, never the whole log.
- **Privacy:** local only, no network, no telemetry.

## Failure modes

1. **Stop hook cannot read the finished reply** (transcript format/timing). This is the #1 risk and the **must-prototype-first** item. If harvesting is unreliable, new terms get silently missed. Mitigation: Plan B below.
2. **Malformed log / json** -> parse defensively, rebuild json from the log, never crash, never block a session.
3. **Two concurrent sessions writing the log** -> atomic writes prevent corruption; worst case one session's new term is missed (recoverable from backups or re-emitted next time). Acceptable for single-user local; documented limitation.
4. **Hook now writes the markdown** -> safer than v2.0's agent-writes-markdown because it is deterministic, atomic, backed-up, and count-guarded.

## Plan B (de-risks the #1 failure mode)

If the Stop-hook harvester proves unreliable, fall back to: **agent keeps appending new terms (v2.0 behavior), but the SessionStart hook runs the normalize + fuzzy-dedup pass** over the log each run. This still delivers the naming-drift fix (the main pain you raised) without depending on Stop-hook harvesting. The agent appending one short line is cheap and was never the context problem (reading the whole log was, and v2.0 already fixed that). So the dedup pass is the irreducible core of v2.1; the harvester is the leanness bonus.

## Deletion test

- Remove the harvester -> v2.0 (agent writes the log). Keep the dedup pass -> still fixes naming drift.
- Remove scheduling -> v1 (discovery only).
- The **normalize + fuzzy-dedup pass is the irreducible win**; the harvester is an additive leanness step gated on the prototype.

## Build order (when we proceed)

1. Prototype the Stop-hook reading the finished reply. Decide harvester vs Plan B.
2. Implement the normalize + fuzzy-dedup module (pure function, unit-tested).
3. Wire it into the chosen writer (harvester or SessionStart).
4. Make the hooks the sole markdown writers; add the count-never-decreases assertion.
5. Update SKILL.md + README + CLAUDE.md; bump to 2.1.0; `claude plugin validate`; test migrate/dedup/graduate/never-shrink.
